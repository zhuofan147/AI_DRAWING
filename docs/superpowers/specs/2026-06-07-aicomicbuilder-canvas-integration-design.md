# AI_DRAWING 画布工作区融合设计方案

## 状态

已通过产品方向审核，进入实现计划准备阶段。

## 背景

在查看 `G:\Project\AIComicBuilder` 后，产品方向从“从零搭建独立桌面画布工具”调整为“以 `G:\Project\AI_DRAWING` 为自有产品主仓库，吸收 AIComicBuilder 的有效业务链路，并新增画布工作区”。

AIComicBuilder 作为参考来源，已经提供了一套较完整的 Next.js 漫剧生成链路，具备以下可借鉴能力：

- 项目管理
- 剧本导入与解析
- 分集管理
- 角色管理
- 分镜生成
- 首尾帧 / 参考帧生成
- 视频提示词生成
- 视频生成
- 视频预览与合成
- 模型配置
- 提示词模板管理
- SQLite + Drizzle 数据持久化
- 本地上传与生成资源管理

因此，AI_DRAWING 不从空白项目开始，也不直接把参考项目原样作为最终产品。迁入后需要裁剪宣传、演示、导出包和无关脚本，只保留可运行的核心业务骨架，再在 AI_DRAWING 中实现自己的画布操作层，帮助用户更直观地查看、组织和触发完整创作流程。

AI-CanvasPro 仅作为画布交互、节点设计、任务状态、模型适配等实现方式参考，不作为直接二次开发代码基座。

## 产品形态

融合后的产品采用“双操作模式”：

- **页面工作台**：适合精细编辑和现有业务流程。
- **画布工作区**：适合全局总览、空间化编排、快捷操作、批量处理、失败重试和跨页面导航。

画布不替代现有页面。画布是同一套项目数据的第二种视图。

## 保留现有页面

以下页面能力继续保留，并作为 AI_DRAWING 的精细操作入口：

- 项目列表
- 剧本 / 导入页面
- 分集管理
- 角色管理
- 分镜列表 / 分镜看板
- 预览与视频合成
- 模型设置
- 提示词模板管理

## 新增画布入口

新增路由：

```text
/[locale]/project/[id]/canvas
```

在项目导航中新增“画布”入口。第一版默认展示项目级画布，后续可以扩展为分集级画布，例如通过 `episodeId` 查询参数或新增分集画布路由实现。

## 信息架构

```text
AI_DRAWING
├─ 页面工作台
│  ├─ 分集

│  ├─ 分镜
```

## 数据映射

画布从 AIComicBuilder 现有数据中派生节点，不新增第二套业务数据。

- `projects` -> 项目节点 `ProjectNode`
- `episodes` -> 分集节点 `EpisodeNode`
- `characters` -> 角色节点 `CharacterNode`
- `storyboardVersions` -> 分镜版本信息或版本选择器
- `shots` -> 镜头节点 `ShotNode`
- `shotAssets` -> 素材节点 `AssetNode`
- `dialogues` -> 镜头节点中的对白 / 配音信息
- 生成动作 -> 操作节点 `ActionNode`
- `finalVideoUrl` 和下载接口 -> 导出节点 `ExportNode`

## 节点类型

### ProjectNode 项目节点

展示项目标题、项目状态、剧本是否存在、分集数量、最终视频状态，以及进入项目相关页面的入口。

### EpisodeNode 分集节点

展示分集标题、序号、生成状态、生成模式、镜头数量、最终视频状态，以及进入分集工作流的入口。

### CharacterNode 角色节点

展示角色名称、角色范围（主角 / 客串）、所属分集、视觉描述、参考图状态，以及跳转角色页面的快捷入口。

### ShotNode 镜头节点

展示镜头序号、镜头描述摘要、帧图状态、视频提示词状态、视频状态、是否过期、是否失败。支持镜头级生成快捷操作。

### AssetNode 素材节点

展示来自 `shotAssets` 的图片、视频或音频素材预览，并能回到所属镜头。

### ActionNode 操作节点

表示可触发的流程动作，例如：

- 拆分分镜
- 批量生成帧图
- 批量生成视频提示词
- 批量生成视频
- 生成参考图
- 合成视频
- 失败重试

### ExportNode 导出节点

提供最终视频下载、项目素材打包下载等入口。

## 连线含义

画布连线表达业务关系，而不是新增业务依赖。

- 项目 -> 分集：项目包含分集。
- 项目 -> 角色：项目拥有全局角色。
- 分集 -> 角色：分集使用某些角色。
- 分集 -> 镜头：分集包含镜头。
- 角色 -> 镜头：角色出现在镜头中，或被镜头素材引用。
- 镜头 -> 素材：镜头产出帧图或视频素材。
- 素材 -> 导出：素材参与预览、合成或下载。

第一版不让用户自由连线来改变业务逻辑。自由连线可以作为后续扩展。

## 画布布局持久化

新增持久化的数据只保存画布布局和 UI 状态，不复制业务数据。

新增表：

```text
canvas_layouts
- id
- projectId
- episodeId nullable
- scope: project | episode
- nodesJson
- edgesJson
- viewportJson
- updatedAt
```

布局数据只保存：

- 节点位置
- 节点是否折叠
- 画布视口
- 用户整理后的视觉状态

布局数据不保存：

- 剧本正文
- 提示词正文
- 生成文件 URL
- API Key
- 模型配置
- 业务状态副本

业务事实始终以数据库中的原始表为准。

## 画布交互设计

第一版画布包含四个区域：

- **左侧资源栏**：按分集、角色、镜头、素材分组展示资源列表。
- **中间画布**：支持拖拽、缩放、平移、自动布局、适应视图。
- **右侧详情栏**：展示选中节点的详情和可执行操作。
- **顶部工具栏**：自动排列、保存布局、只看失败、只看未完成、一键续跑。

## 节点状态颜色

- 灰色：未开始
- 蓝色：可生成
- 黄色：生成中
- 绿色：已完成
- 红色：失败，或内容过期且阻塞后续生成

## 节点快捷操作

节点上可提供以下操作：

- 打开详情
- 跳转到原页面
- 生成 / 批量生成
- 失败重试
- 复制提示词
- 预览素材
- 下载结果

## 页面与画布同步规则

页面和画布必须共用同一套数据来源和 API。

同步规则：

- 页面操作完成后，画布刷新派生状态。
- 画布操作调用现有页面已经使用的 API。
- 画布操作完成后，页面也能看到结果。
- 画布只保存布局，不保存业务副本。
- 用户永远可以从画布节点一键跳回原页面。

## 第一版可触发的生成动作

画布第一版可以调用现有生成接口中的这些动作：

- `shot_split`
- `single_frame_generate`
- `batch_frame_generate`
- `single_video_prompt`
- `batch_video_prompt`
- `single_video_generate`
- `batch_video_generate`
- `single_reference_video`
- `batch_reference_video`
- `video_assemble`

画布调用这些动作时，应传入现有页面已经使用的字段，例如：

- `modelConfig`
- `episodeId`
- `versionId`
- `shotId`
- `ratio`
- `overwrite`

画布不重新实现生成逻辑。

## 实现边界

第一版必须实现：

- 新增画布路由。
- 新增项目导航入口。
- 新增 `canvas_layouts` 表和迁移。
- 新增画布布局 GET / PUT API。
- 新增业务数据到画布节点的映射方法。
- 新增画布 UI。
- 新增节点状态计算。
- 新增核心节点操作，并复用现有 API。
- 增加映射、状态、布局持久化、操作请求构造相关测试。

第一版不做：

- 不重写 AIComicBuilder 的生成流程。
- 不替代分镜页、角色页、分集页或预览页。
- 不新增第二套项目模型。
- 不做桌面端打包。
- 不做多人协作。
- 不做完整插件系统。
- 不让任意画布连线直接改变业务逻辑。

## 验收标准

融合完成后，应满足：

- 用户可以打开一个已有 AIComicBuilder 项目，并进入画布页。
- 画布能展示项目、分集、角色、镜头、活跃素材和导出节点。
- 用户无需离开画布，也能理解项目到分集、角色、镜头、素材、导出的关系。
- 拖动节点后保存布局，刷新页面后位置仍保留。
- 自动布局可以恢复清晰可读的画布。
- 可以筛选失败节点和未完成节点。
- 选中节点后，右侧详情栏展示相关信息和操作。
- 从节点可以跳转回对应原页面。
- 镜头节点可以通过现有生成接口触发一次生成动作。
- 分集节点或操作节点可以触发批量生成动作。
- 现有分集、角色、分镜、预览页面仍能正常使用。

## 风险与应对

### 生成接口复杂

`src/app/api/projects/[id]/generate/route.ts` 已经承载大量动作。画布应该调用它，而不是复制生成逻辑。

### 项目级和分集级数据形状不同

项目 GET 和分集 GET 聚合数据的方式不完全一样。画布映射层需要统一处理，避免 UI 关心接口差异。

### 布局可能变脏

当分集、角色、镜头或素材被删除后，旧布局里可能还保留对应节点位置。映射层需要合并新业务数据和旧布局，并丢弃不存在实体的布局项。

### 画布可能信息过载

复杂项目节点很多。第一版必须提供筛选、分组、自动布局和资源栏，避免画布变成一团乱线。

### 画布依赖增加包体

如果使用 React Flow，会增加前端依赖。应把画布代码隔离在 canvas 路由和相关组件内，不影响其他页面。

## 待实现阶段确认的问题

- 第一版只做项目级画布，还是同时做分集级画布？
- 画布是否使用 React Flow？当前实现计划建议使用 React Flow，因为它能快速提供拖拽、缩放、小地图、连线和适应视图。
- `canvas_layouts` 第一版是否同时支持 project 和 episode 两种 scope？当前设计建议表结构预留两种 scope，但第一版页面先做 project scope。
